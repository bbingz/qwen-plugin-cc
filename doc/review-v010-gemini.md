- Top-level counts: 2 P0, 2 P1, 2 P2
- Key drift one-liner: While the implementation robustly incorporates real-world CLI nuances, the architectural spec and implementation plan have fallen behind the codebase, leaving unfinished plan steps, unchecked checkboxes, and outdated schema definitions.

## P0 (blockers / must-fix before wider release)

### P0-1: Plan completely lacks Phase 3-5 and retains unchecked boxes
- **Category**: plan.md archivability (Target 6)
- **Where**: `docs/superpowers/plans/2026-04-20-qwen-plugin-cc-implementation.md`
- **Problem**: The plan abruptly ends after Task 2.18. Detailed tasks for Phase 3 (Review), Phase 4 (status/cancel/hooks), and Phase 5 (Polish & Docs) are missing entirely. Furthermore, all checkboxes for Phase 0-2 tasks remain unchecked (`- [ ]`), so the plan cannot serve as an accurate historical archive of completed work.
- **Fix**: Append the Phase 3-5 tasks to the plan, and mark all completed tasks from Phase 0-5 with `- [x]`.

### P0-2: Spec §3.3 contradicts actual `auto-edit` background behavior
- **Category**: spec vs code drift (Target 1)
- **Where**: `docs/superpowers/specs/2026-04-20-qwen-plugin-cc-design.md` Section 3.3
- **Problem**: Spec §3.3 explicitly states that background tasks without `--unsafe` will be blocked and return `require_interactive`. However, Phase 0 finding F-13 and the `qwen-companion.mjs` code allow background tasks without `--unsafe` to run successfully using the `auto-edit` default (which merely results in shell tool `permissionDenials` instead of failing). The `require_interactive` error is only thrown for explicit `yolo` usage without `--unsafe`.
- **Fix**: Update Spec §3.3 and §5.2 error tables to reflect that background jobs default to `auto-edit` and succeed (producing `permissionDenials`), whereas `require_interactive` strictly guards explicit `yolo` mode in the background.

## P1 (should-fix soon)

### P1-1: Job schema fields are mismatched or unused in code
- **Category**: job schema field audit (Target 7)
- **Where**: `docs/superpowers/specs/2026-04-20-qwen-plugin-cc-design.md` Section 4.6 vs `qwen-companion.mjs`
- **Problem**: The spec defines `logPath` and `phase` in the job schema. However, `qwen-companion.mjs` never uses or writes `phase`, and it writes the log path under the `logFile` key instead of `logPath`.
- **Fix**: Update the job schema documentation in §4.6 to use `logFile` and remove the unused `phase` field, aligning it perfectly with the JSON payloads generated by `qwen-companion.mjs`.

### P1-2: `id` vs `jobId` split remains as technical debt
- **Category**: FINDINGS.md current status (Target 4)
- **Where**: `doc/probe/FINDINGS.md` F-17 & `qwen-companion.mjs`
- **Problem**: F-17 correctly identifies the `id` vs `jobId` field split inherited from the Gemini plugin. While mitigated by fallback logic (`j.jobId ?? j.id`) in `qwen-companion.mjs` and `state.mjs`, relying on this coexistence is brittle and complicates state interactions moving forward.
- **Fix**: Formally deprecate `id` in favor of `jobId`, write a tiny migration function in `state.mjs` to rewrite older job files if necessary, and remove the `j.jobId ?? j.id` coalesce checks.

## P2 (nice-to-have / polish)

### P2-1: Flag mismatch in `rescue.md` argument hint
- **Category**: commands doc accuracy (Target 2)
- **Where**: `plugins/qwen/commands/rescue.md`
- **Problem**: The command frontmatter `argument-hint` shows `[--resume|--fresh]`. While the `qwen-rescue` subagent correctly handles and translates `--resume` into `--resume-last` for the backend, `qwen-companion.mjs` only parses `--resume-last`. This discrepancy could confuse developers invoking the companion script manually.
- **Fix**: Add `--resume` as a direct alias for `--resume-last` in the `parseArgs` configuration within `qwen-companion.mjs`.

### P2-2: README claims "Phase 0-4 done" but includes Phase 5 work
- **Category**: README + lessons.md claims vs reality (Target 5)
- **Where**: `README.md` Status section
- **Problem**: The README states "Status: v0.1 complete (Phase 0-4 done...)". However, writing the `README.md` and `lessons.md` are explicitly Phase 5 tasks in the project plan, meaning Phase 5 is effectively complete as well.
- **Fix**: Change the status text in the README to say "Phase 0-5 done" to reflect that the polish and documentation phase has also been finalized.
