**动机**:gemini baseline §6.3 靠 `stats.models` 区分 requested vs. served model,判断 qwen CLI 是否吐等效字段,决定 Phase 2 接通还是删 timing 基础设施。

- [ ] **Step 1: 写 probe 脚本**

Create `doc/probe/case-14-result-event.sh`:

```bash
#!/usr/bin/env bash
# Case 14: qwen CLI result event 原始 schema
# 目的:抓 stream-json 最后的 result event 结构,看是否有 stats.models
#       以及 input/output/thoughts token 字段,决定 v0.3 timing 方向。
set -uo pipefail
cd "$(dirname "$0")"

raw=$(qwen "reply with exactly: ready" \
  --output-format stream-json --max-session-turns 1 2>&1)
exit_code=$?

# 提取所有 JSON 行里 type==result 的那条
result_line=$(printf '%s\n' "$raw" | awk 'BEGIN{RS="\n"} /^\{/ {print}' | \
  jq -c 'select(.type=="result")' 2>/dev/null | head -1)

if [ -z "$result_line" ]; then
  result_line=$(printf '%s\n' "$raw" | grep -E '^\{.*"result"' | head -1)
fi

keys=$(printf '%s' "$result_line" | jq -r 'paths(scalars) | map(tostring) | join(".")' 2>/dev/null | sort -u | tr '\n' ',')

jq -n \
  --arg cmd 'qwen "reply with exactly: ready" --output-format stream-json --max-session-turns 1' \
  --arg raw "$raw" \
  --arg result "$result_line" \
  --arg keys "$keys" \
  --argjson exit_code $exit_code \
  '{case:"14-result-event",
    cmd:$cmd,
    exit_code:$exit_code,
    raw_tail:($raw|.[-2048:]),
    result_event:$result,
    scalar_key_paths:$keys}' \
  > case-14-result-event.result.json

echo "exit_code: $exit_code"
echo "--- result_event ---"
echo "$result_line"
echo "--- scalar key paths ---"
echo "$keys"
```

**动机**:gemini baseline §1 的 `/gemini:ask` 是 streaming / 前台阻塞 / 不持久化(设计上),Qwen 侧之前靠 `/qwen:rescue`(不带 --background)覆盖该 UX,但命令名错位 → 跨 provider 肌肉记忆断裂。v0.3 补命令名对齐,不改 companion router。

**语义**:复用 `task` sub,固定 `background=false, resume-last=false, fresh=true implicit, unsafe=false`。用户要复杂语义仍走 `/qwen:rescue`。

- [ ] **Step 1: 写 command markdown**

Create `plugins/qwen/commands/ask.md`:

```markdown
---
description: Ask Qwen a one-shot question — foreground streaming, no persistence, no session resume
argument-hint: '[--model <model>] [what you want to ask Qwen]'
allowed-tools: Bash(node:*)
---

One-shot foreground ask to Qwen。轻量、前台阻塞、不 resume、不后台。

需要 resume 上次 / 后台跑 / 让 qwen 实际执行 shell/write 的场景请用 `/qwen:rescue`。

Raw user request:
$ARGUMENTS

## Execution

单次 Bash,直接 stream qwen 输出给用户,不做任何加工:

```bash
node "${CLAUDE_PLUGIN_ROOT}/scripts/qwen-companion.mjs" task "$ARGUMENTS"
```

**参数处理**:

- `--model <model>` → 保留传给 task
- 其他 flag(`--background` / `--wait` / `--unsafe` / `--resume-last` / `--fresh` / `--effort`)→ 不支持,若用户传了,提示改用 `/qwen:rescue`
- 没有 prompt → 问用户"What do you want to ask Qwen?"

## Operating rules

- 不调 resume-candidate 探活,也不 ask user continue/fresh — ask 永远是新对话
- 不 poll status、不 fetch result、不 cancel
- 返回 companion stdout 原样,不加前后注释
- 如 companion 报缺 qwen / 未授权,提示 `/qwen:setup`

## 与 `/qwen:rescue` 的分工

| 场景 | 用这个 |
|---|---|
| 问一句话、看眼输出 | `/qwen:ask` |
| 委派调研/修代码,可能 > 2 分钟 | `/qwen:rescue --background` |
| 要 qwen 真跑 shell/edit | `/qwen:rescue --unsafe` |
| 续上次 qwen 对话 | `/qwen:rescue --resume-last` |
```
